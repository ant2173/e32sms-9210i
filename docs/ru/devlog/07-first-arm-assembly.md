# Соник на Nokia 9210i, часть 7: первый ARM-ассемблерный блиттер

**Дневник разработки — 3 июля 2026 года**  
[English version](../../en/devlog/07-first-arm-assembly.md)  
[← Часть 6: Из слайд-шоу — в движение](06-from-slideshow-to-motion.md)

Предыдущий этап оптимизации закончился на чёткой границе. Frameskip ускорил игровую логику, парная 32-битная запись уменьшила число обращений к framebuffer, а оставшиеся горячие участки уже представляли собой вполне приличный C-код. Следующий заметный прирост требовал не очередного флага компилятора и не небольшой правки цикла, а ARM-ассемблера.

В такой момент велик соблазн сразу написать большой блиттер. Это же самый быстрый способ потратить вечер на борьбу с древним тулчейном. Поэтому первым шагом стал не блиттер, а минимальная проверка: способен ли SDK9210 вообще собрать ассемблерный файл, связать его с программой и правильно вызвать на настоящем телефоне.

## Сначала доказать сам механизм

Основания для осторожности были. Порт SNES9x для UIQ, который использовался как архитектурный ориентир, не мог штатно собирать свои ассемблерные исходники. Система сборки переводила расширение `.S` в нижний регистр, из-за чего ломалась предварительная обработка, и автору пришлось применять отдельный Makefile.

В E32SMS такой обходной путь не понадобился.

Я добавил крошечный `asmtest.S` с двумя намеренно скучными функциями:

- `asm_add(40, 2)`, которая должна вернуть 42;
- `asm_fill16(...)`, которая должна заполнить небольшой массив заранее известными значениями.

Функции были объявлены с C-совместимым связыванием, добавлены в обычный `.mmp` и собраны стандартной цепочкой `bldmake` / `abld`. Сборка ARMI UREL запустилась на Nokia 9210i и записала в лог правильные результаты.

Этот тест сразу доказал четыре вещи:

1. SDK9210 принимает препроцессируемый файл `.S` через штатную систему сборки;
2. объектный код ассемблера нормально линкуется с C- и C++-частями E32SMS;
3. соглашение о вызовах работает ожидаемым образом;
4. отдельный линкер и самодельный сценарий сборки не нужны.

Только после этого началась настоящая оптимизация.

## Почему первым стал именно вывод в видеопамять

Переписывать первым делом Z80 или фоновый рендерер было бы слишком рискованно. Там сложное состояние, ветвления и множество взаимосвязанных правил. Гораздо безопаснее оказался финальный цикл преобразования индексированных пикселей в формат экрана: небольшой изолированный участок, написанный специально для E32SMS и уже хорошо измеренный профилировщиком.

Для каждого выводимого пикселя C-код выполнял одну и ту же последовательность:

1. прочитать 8-битный индекс цвета;
2. ограничить его активным диапазоном из 32 цветов;
3. взять из палитры 16-битное значение RGB444;
4. упаковать соседние пиксели;
5. записать результат в framebuffer Nokia.

Предыдущая C-оптимизация уже объединяла два 16-битных пикселя в одну 32-битную запись. Ассемблерный вариант идёт дальше: подготавливает четыре пикселя как два машинных слова и выгружает оба слова одной командой множественной записи ARM:

```asm
stmia r0!, {r4, r7}
```

Публичный C-интерфейс при этом остаётся простым:

```c
void asm_remap_line(
    unsigned short *dst,
    const unsigned char *src,
    const unsigned short *palette,
    int pixel_count
);
```

Ассемблерная ветка включается макросом `E32_ASM_REMAP`. Старый C-вариант с парной записью сохранён как мгновенный fallback. Эксперимент остаётся обратимым, а не превращается в безвозвратное переписывание рабочего кода.

## Три ловушки двадцатилетнего тулчейна

Первые сборки упали по трём совершенно разным причинам. Теперь все три известны и записаны.

### 1. Комментарии ассемблера видит C-препроцессор

Файл `.S` сначала проходит через препроцессор и лишь затем попадает в ассемблер. Апостроф в английском комментарии был воспринят как начало символьной константы и дал ошибку `unterminated character constant`.

Практическое правило: в комментариях к ассемблерным исходникам под SDK9210 лучше не использовать апострофы и кавычки.

### 2. Полусловная загрузка ARMv4 не поддерживает масштабированный индекс

Первая версия обращения к палитре пыталась сделать примерно следующее:

```asm
ldrh rX, [rPalette, rIndex, lsl #1]
```

Для `ldrh` такой режим адресации на этой архитектуре недоступен. Индекс нужно сначала удвоить отдельной арифметической инструкцией, а затем использовать как обычное регистровое смещение.

Обычные загрузки слова и байта здесь гибче; загрузка полуслова — нет.

### 3. `abld clean` не удаляет все сгенерированные файлы

После удаления тестового исходника сборка продолжала его искать. В дереве `Epoc32\BUILD\...\GROUP_EMU\` сохранился старый список файлов.

Потребовалось вручную удалить этот сгенерированный каталог, после чего снова выполнить `bldmake` и `abld`. Только тогда метаданные проекта были построены заново.

## Измеренный результат

Изображение после перехода на ассемблер осталось полностью идентичным C-версии: без сдвигов строк, ошибок палитры и искажений геометрии. Для цикла, который проходит через каждый отображаемый пиксель, это принципиально важная проверка.

Разница в скорости получилась существенной:

| Сборка | Преобразование и вывод в VRAM | Логика игры | Выведенные кадры |
|---|---:|---:|---:|
| C с парной записью | ~14,8 мс | ~24,5 fps | ~12,2 fps |
| ARM-ассемблер | ~8,2 мс | ~27,0 fps | ~13,5 fps |

Этап преобразования и вывода ускорился примерно в **1,8 раза**, а его время сократилось примерно на **44%**. Общая скорость игровой логики выросла примерно на **10%**, частота реально выводимых кадров — примерно на **11%**.

Локальное ускорение в 1,8 раза не превращается в такое же ускорение всей программы: блиттер является только одной частью кадра. Закон Амдала действует даже на Nokia Communicator.

## Новая карта кадра

При `frameskip = 1` один реально отрисованный кадр теперь примерно распределяется так:

| Этап | Время |
|---|---:|
| Фоновый рендеринг | ~11,5 мс |
| Рендеринг спрайтов | ~8,8 мс |
| Ассемблерное преобразование и вывод в VRAM | ~8,2 мс |
| Работа с кэшем тайлов | ~2,6 мс |
| Остальные расходы рендера | несколько миллисекунд |
| **Всего на отрисованный кадр** | **~34 мс** |

Преобразование framebuffer больше не является крупнейшим измеренным компонентом. На первое место вышел фоновый рендерер.

За весь цикл профилирования и оптимизаций скорость игровой логики выросла примерно с **12,5 до 27,0 fps**. Эти цифры не равны частоте изображения, потому что текущая сборка использует frameskip. Но игра теперь продвигается более чем вдвое быстрее исходной измеренной версии и достигает примерно **45% от 60-герцового темпа Master System**.

## Что на самом деле доказал этот этап

Главный результат шире одного ускоренного цикла.

Полный ассемблерный путь теперь подтверждён на настоящем устройстве:

- исходники `.S` собираются штатным проектом SDK9210;
- ассемблерные функции линкуются и вызываются из C;
- соглашение о вызовах ARM работает;
- пакетная запись в физический framebuffer даёт правильные пиксели;
- C-вариант остаётся для сравнения и быстрого отката;
- прирост подтверждён тестом на реальной Nokia 9210i.

Это открывает путь к более сложной работе. Очевидные следующие цели — фон и спрайты, но они намного труднее: там атрибуты тайлов, прокрутка, приоритеты и гораздо больше ветвлений. Финальный вывод изображения был правильным первым объектом для ассемблерной оптимизации.

## Следующая полезная функция — уже не новый benchmark

Управления по-прежнему нет. Соник движется уже достаточно бодро, чтобы отсутствие кнопок раздражало сильнее, чем оставшийся недостаток скорости. Пока телефон показывает встроенную демку игры, а не позволяет человеку играть.

Поэтому следующий самый ценный шаг — связать клавиатуру Nokia с портами контроллера Master System. Новые ассемблерные оптимизации можно продолжить после того, как эмулятор перестанет быть только демонстрацией и станет управляемым.

Продолжение следует.
