# ARM-ассемблерный блиттер преобразования и вывода в VRAM

[English version](../../en/engineering/arm-assembly-blitter.md)

## Предмет отчёта

Этот документ описывает первую ARM-ассемблерную оптимизацию E32SMS: замену
C-цикла преобразования индексированных цветов в RGB444 и записи в framebuffer
функцией `asm_remap_line`.

Изменение собрано в конфигурации ARMI UREL штатной цепочкой SDK9210
`bldmake` / `abld` и проверено на оригинальной Nokia 9210i.

## Исходное состояние

Конфигурация теста:

- `frameskip = 1`;
- прямой доступ к физическому framebuffer с проверкой фокуса;
- изображение SMS 256×192 по центру экрана 640×200 Color4K;
- резервный C-вариант упаковывает два пикселя RGB444 в одну 32-битную запись.

Измеренная исходная версия:

| Показатель | Значение |
|---|---:|
| Преобразование и вывод в VRAM | ~14,8 мс на отрисованный кадр |
| Скорость логики | ~24,5 fps |
| Частота вывода | ~12,2 fps |

## Проверка пути сборки

Перед интеграцией блиттера файл `asmtest.S` доказал, что исходник `.S`,
перечисленный в `.mmp`, может быть:

1. обработан и собран тулчейном SDK9210;
2. слинкован с существующим EXE;
3. вызван через C-совместимый интерфейс;
4. корректно выполнен на ARM-железе.

Аппаратный тест проверял возврат простого значения и запись в небольшой массив.

## Интерфейс

```c
void asm_remap_line(
    unsigned short *dst,
    const unsigned char *src,
    const unsigned short *palette,
    int pixel_count
);
```

Реализация выбирается макросом `E32_ASM_REMAP`. При его отключении используется
предыдущий C-цикл.

## Путь данных

Для каждого пикселя:

1. загрузить 8-битный индекс;
2. применить маску `0x1F`;
3. масштабировать индекс для обращения к 16-битной палитре;
4. загрузить RGB444 инструкцией `ldrh`;
5. объединить два 16-битных пикселя в одно 32-битное слово;
6. подготовить два слова и записать четыре пикселя командой:

```asm
stmia r0!, {r4, r7}
```

Выравнивание адреса назначения и ширина строки 256 пикселей позволяют
обрабатывать строку группами по четыре пикселя.

## Измеренный результат

| Показатель | C-вариант | Ассемблер | Изменение |
|---|---:|---:|---:|
| Преобразование и вывод в VRAM | ~14,8 мс | ~8,2 мс | -44%, примерно в 1,8 раза быстрее |
| Скорость логики | ~24,5 fps | ~27,0 fps | +10% |
| Частота вывода | ~12,2 fps | ~13,5 fps | +11% |

На устройстве изображение осталось визуально идентичным: цвета, положение строк
и геометрия кадра не изменились.

## Обнаруженные особенности ассемблера SDK9210

### Предварительная обработка `.S`

Перед ассемблированием работает C-препроцессор. Апострофы и кавычки в
комментариях могут быть восприняты как незакрытые токены. Комментарии лучше
делать предельно простыми.

### Адресация `ldrh` на ARMv4

Использованный вариант загрузки полуслова не допускает масштабированного
регистрового смещения:

```asm
ldrh rX, [rBase, rIndex, lsl #1]   /* здесь недопустимо */
```

Индекс нужно предварительно умножить отдельной инструкцией, после чего применять
обычное регистровое смещение.

### Старый список исходников в сгенерированном дереве

`abld clean` не удалил метаданные `Epoc32\BUILD\...\GROUP_EMU\`. После исключения
`asmtest.S` из проекта сборка продолжала ссылаться на него. Проблема исчезла
после ручного удаления сгенерированного каталога и повторного создания build-файлов.

## Текущий профиль

Примерное время одного реально отрисованного кадра:

| Компонент | Время |
|---|---:|
| Фоновый рендеринг | ~11,5 мс |
| Рендеринг спрайтов | ~8,8 мс |
| Ассемблерное преобразование и вывод | ~8,2 мс |
| Кэш тайлов | ~2,6 мс |
| Весь рендеринг | ~34 мс |

Ассемблерный блиттер больше не является крупнейшим измеренным компонентом.
Следующими целями для серьёзной оптимизации могут стать фон и спрайты, но их
логика существенно сложнее.

## Указание источника идеи

Общий подход с пакетной ARM-записью при прямом выводе в framebuffer раннего
Symbian был сформирован после изучения порта SNES9x UIQ от notaz. В этой
реализации не предполагается включение чужого исходного кода; порядок указания
влияний и сторонних компонентов описан в документах репозитория.
